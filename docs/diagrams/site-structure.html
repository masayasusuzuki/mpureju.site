テキスト相談フロー（Phase 3）
構造化データ（料金・施術名）→ SQL検索で正確に取得
非構造化データ（コラム・施術詳細）→ pgvector RAGで意味検索
DeepSeek APIがクエリ分解・応答生成に専念。データ検索はコード側が担当。
① ユーザーが質問を送信
チャットウィンドウからテキストを送信すると /api/chat に POST される。
② DeepSeek 1回目：意図分類 + クエリ分解
ユーザーの口語表現を検索に適したキーワードに変換。同時に「料金・施術・FAQ・コラム・挨拶」のどれかに意図を分類し、呼び出すツールを決定する。
③ コード側でデータ検索
ツールの種類に応じて Supabase SQL（料金・施術）またはpgvectorベクトル検索（コラム・ナレッジ）を実行。LLMにはURLを生成させず、コードが責任を持って正確なデータを取得する。
④ DeepSeek 2回目：応答生成
検索結果をコンテキストとして渡し、自然な日本語で最終回答を生成。会話履歴は chat_messages テーブルに保存し、連続した会話が可能。

写真でおすすめ診断フロー（Phase 3）
画像解析 → Gemini API（マルチモーダル）／テキスト解釈 → DeepSeek API
写真は保存しない（base64でそのまま送信・利用規約に明記）
診断実行条件：写真アップロード AND（テキスト入力 OR タグ選択）の両方が必須
① 入力バリデーション
写真と悩み（テキスト or タグ）の両方が揃うまで「AIに診断してもらう」ボタンは disabled のまま。片方だけではAPIを呼ばない設計。
② Gemini API で画像解析
写真をbase64に変換してGemini APIへ送信。美容医療の観点から顔・肌の状態を分析したテキストを返す。写真はサーバーに一切保存しない。システムプロンプトはSupabaseの chatbot_prompts で管理・編集可能。
③ DeepSeek 1回目：クエリ分解
Geminiの画像解析テキスト + ユーザーの悩みテキストを統合。「どの部位（pillar）か」「どんなキーワードで施術・記事を検索すべきか」をJSON形式で出力させる。
④ microCMS でコンテンツ検索
DeepSeekが出力したキーワードをもとにコード側でmicroCMS treatments / columns を検索。マッチした施術・記事のURLをリンクとして取得。LLMにURLを生成させないことでハルシネーションを防ぐ。
⑤ DeepSeek 2回目：最終アドバイス生成
画像解析結果・ユーザー悩み・検索結果をすべて統合し、自然な日本語でアドバイス文を生成。UIには「アドバイステキスト」「おすすめ施術リンクカード」「おすすめ記事リンク」の3セクションで表示する。

ナレッジ登録フロー
テキスト相談でコラム・施術詳細などの「詳しい情報」に答えるためのRAGナレッジを管理画面から登録・編集する仕組み。
登録するだけで自動的にベクトル化され、チャットボットがすぐ参照できるようになる。
① 管理画面でナレッジを登録・編集
mpureju.admin の管理画面から chatbot_knowledge テーブルにコンテンツを追加・編集する。施術詳細・よくある質問・クリニック情報などを自由記述で登録可能。
② Supabase Edge Function が自動起動
DB保存をトリガーにEdge Functionが起動。OpenAI Embedding APIにテキストを送り、意味ベクトルを生成する。
※ Embeddingのみ OpenAI。チャット応答はDeepSeek。
③ knowledge_embeddings にupsert → 即座に検索可能
生成したベクトルを knowledge_embeddings テーブルにupsert。テキスト相談でユーザーが質問すると、pgvector のコサイン類似度検索でこのナレッジが参照される。登録した瞬間からチャットボットに反映される。

🟢 緑 = Supabase DB / 管理画面　　🟣 紫 = フロントエンドページ　　🔵 青 = 管理画面操作
このサイトは 「どこかを変えたら別の場所も直さなければならない」依存関係がほとんどない 設計になっています。
コンテンツ・料金・ページ・AI知識——それぞれが独立して拡張できます。
Before / After — ハードコード vs SSOT設計
❌ Before — WordPress ハードコード
  • 料金変更 → どのファイルを直せばいいか不明
  • 変更漏れがあっても検知できない
  • 施術追加 → コードとCMSを両方手動で同期
  • 254ページ、管理者以外は触れない
  • AI連携しようとするとデータ形式がバラバラで詰む
✅ After — SSOT設計（現サイト）
  • 料金変更 → 管理画面で1行編集、全ページ自動反映
  • 施術追加 → microCMSに登録するだけでページが生える
  • AIチャットボットもDBを直接参照 → 同期不要
  • スタッフが管理画面から安全にCRUD操作
  • データフローが1本化されているため変更漏れが構造上起きない
拡張パターン — 何をしたいときに何を触るか
やりたいこと 触る場所 コード変更 自動反映される箇所
料金を変更する 管理画面 / price_items 不要 /price/ ・施術詳細サイドバー・内服薬詳細・AIチャットボット
施術一覧に新施術を追加 管理画面 / treatment_items 不要 /treatment/ ・AIチャットボット
施術詳細ページを追加 microCMS / treatments 不要 /{pillar}/{slug}/ が自動生成（ISR）
新しいピラーページを追加 page.tsx 約50行追加 最小限 TreatmentDetailTemplate がそのまま適用される
AIの知識を追加・更新 管理画面 / chatbot_knowledge 不要 Embedding自動生成 → RAG検索に即反映
AIの回答トーンを変える 管理画面 / chatbot_prompts 不要 次のチャットから即反映
内服薬・機器を追加 microCMS / medicines or machines 不要 /medicine/[slug]/ か /machine/[slug]/ が自動生成
新コンテンツ種別を追加
（例: 用語集・比較記事）		 microCMS にAPI追加 ページ1枚のみ 以降はCMS管理だけで運用可能
コンバージョン計測を追加 conversion_events に event_type 追加 enum 1行のみ 蓄積データが分析基盤に自動反映
拡張の独立性 — どのレイヤーも他に波及しない

設計のポイント — なぜ拡張コストが低いか
🗂 データの役割分担が明確
📝
microCMS編集者が書くリッチコンテンツ（施術説明・画像・コラム）
🗄
Supabase構造化データ（料金・施術名・問い合わせ・AIログ）
役割が明確なので「どちらに入れるか迷う」が起きない。新データも自然に配置先が決まる。

料金・施術データがSupabaseにあるため、AIが常に最新データを参照

通常のAIチャットボットは知識を再学習させる必要がある。このサイトはDBがそのまま知識ソース。

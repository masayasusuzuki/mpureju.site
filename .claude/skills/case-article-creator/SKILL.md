# Case Article Creator

Instagram症例投稿の画像とキャプションから、microCMS casesスキーマに準拠した症例記事（article.md）を生成するスキル。

---

## Instagram URL管理

症例記事のInstagram投稿URLは `public/case/urls.txt` で管理する。

- 1行1件でURLを貼り付ける
- 処理済みの行は先頭に `# [完了]` を付けてコメントアウト
- `public/case/` 配下の各ディレクトリ番号と、urls.txtの上から順に1対1で対応する

### URL整合性ルール（過去のバグ教訓）

過去にバッチ処理の再開時に処理済みURLを二重処理し、dir 055 以降の全記事で instagram_url がズレるバグが発生した（詳細: `docs/CMS_FIX_PLAN.md`）。再発防止のため以下を厳守する：

1. **ディレクトリ番号 = urls.txt の行番号（Instagram URL のみカウント）** であることを必ず検証する
2. article.md 作成時、instagram_url はディレクトリ番号に対応する urls.txt の行から取得する。キャプションや他のソースから推測しない
3. バッチ処理の再開時は、**最後にリネーム済みのディレクトリ番号**を確認し、その次の番号から再開する。urls.txt の `# [完了]` マーカーだけに頼らない
4. 投稿前に microCMS に同じ slug が既に存在しないか確認する（重複投稿防止）

## ワークフロー

### Step 1: 対象ディレクトリの特定

ユーザーが指定したディレクトリ（例: `public/case/001/`）を確認する。
指定がない場合は `public/case/` 配下で `article.md` がまだないディレクトリを探す。

Instagram URLは `public/case/urls.txt` から対応するURLを取得する。

### Step 2: 画像とキャプションを読み込む

**画像：** ディレクトリ内の画像ファイル（.jpg, .jpeg, .png）をすべて番号順に読み込む。
症例写真から以下を把握する：
- 施術部位（目元・鼻・口元・フェイスライン等）
- 施術内容の手がかり（Before/Afterの変化）
- 経過タイミング（直後・1週間後等）

**キャプション（caption.txt）：** Instagramの投稿コメント原文。
施術名・部位・効果・リスク等の情報ソースとして使用する。
口語的な文体はそのまま使わず、症例記事として読みやすく整える。

### Step 3: pillar（部位カテゴリ）を判定する

**⚠️ 最重要ルール：pillarは施術名（treatment_label）に基づいて判定する。施術部位や効果で判断してはいけない。**

microCMS `treatments` APIに登録されている施術→pillarの対応関係に従う。以下が正式なマッピングである：

#### mouth（口元）
口角挙上、口唇拡大、口唇縮小、外側人中短縮、人中短縮、ピーナッツリップ、M字リップ、ガミースマイル手術

#### eye（目元）
二重埋没、二重抜糸、二重全切開、眼瞼下垂、眉下切開、目頭切開、目頭上切開、脱脂、上眼瞼除皺、下眼瞼除皺、下眼瞼除皺＋脱脂、脂肪再配置、表ハムラ、裏ハムラ、目尻切開、グラマラス（グラマラスライン形成）、蒙古襞形成

#### nose（鼻）
プロテーゼ、鼻尖形成、耳介軟骨移植、ストラット法オプション、鼻中隔軟骨移植、肋軟骨移植、鼻中隔延長、鼻翼縮小、鼻孔縁下降、鼻翼挙上、鼻孔縁挙上、鼻骨骨切り、ハンプ切除、鼻翼基部形成（貴族）、鼻柱基部形成（猫手術）、側頭筋膜移植、プロテーゼ抜去

#### lift（リフトアップ）
糸リフト、ショートスレッド、アイスレッド

#### skin（美容皮膚科）
ヒアルロン酸注射、ボトックス注射、ソフウェーブ、ブレッシング、メソナJ、ウルトラセルZi(HIFU)、ザーフ（RF）、サーマジェン（RF）、サーマZi(RF＋HIFU)、ポテンツァ、サーマニードル（RF）、カスタマイズレーザー（レーザー治療）、IPL/フォトフェイシャル、KOライト、スキンブースター、脂肪溶解注射、ピーリング、ほくろ/イボ除去（CO2レーザー治療）、ほくろ/イボ除去（切除）

**判定手順：**
1. キャプションと画像から施術名を特定する
2. 上記マッピングで該当するpillarを決定する
3. **マッピングに該当施術がない場合**は、ユーザーに確認してから決定する

**よくある間違い（絶対にやらないこと）：**
- ❌ 「唇にヒアルロン酸 → 口元ピラー」 → ⭕ ヒアルロン酸注射は **skin**
- ❌ 「目の下にボトックス → 目元ピラー」 → ⭕ ボトックス注射は **skin**
- ❌ 施術部位でpillarを推測する → ⭕ 必ず施術名でマッピングを引く

pillarの日本語表記：
- `mouth` → `"口元"`
- `eye` → `"目元"`
- `nose` → `"鼻"`
- `lift` → `"リフトアップ"`
- `skin` → `"美容皮膚科"`

### Step 4: CASE_INDEX.md で次の番号を確認する

`public/case/CASE_INDEX.md` を読み込み、判定したpillarの次の連番を取得する。
slug命名規則: `{pillar英語名}-{連番3桁}`（例: `skin-001`, `mouth-002`）

インデックスの該当行を更新してから記事を作成する（番号の重複を防ぐため）。

### Step 5: frontmatter を設定する

microCMS casesスキーマに準拠したYAML frontmatterを生成する。

```yaml
---
title: "記事タイトル（施術内容と効果がわかるSEO向け日本語、30〜50文字）"
slug: "{pillar英語名}-{連番3桁}"
pillar:
  - "日本語pillar名"
treatment_label: "施術名（microCMS treatmentsに登録されている正式名称）"
timing: "経過タイミング（直後/1週間後/1ヶ月後 等）"
concern: "お悩みキーワード（カンマ区切り）"
risks: "リスク・副作用（医療広告GL必須）"
tags: "検索用タグ（カンマ区切り）"
thumbnail: "image_1.jpg"
instagram_url: "元のInstagram投稿URL"
published_at: "YYYY-MM-DD"
---
```

#### フィールド詳細

- **treatment_label**: microCMS treatmentsに登録されている正式名称を使用。キャプションの表記ゆれ（例:「ヒアルロン酸」→「ヒアルロン酸注射」）は正式名称に揃える
- **timing**: キャプションや画像から特定。不明な場合は空文字
- **concern**: 患者の悩みをカンマ区切りで。キャプションのハッシュタグも参考にする
- **risks**: キャプションの「リスク副作用」記載をそのまま使用。なければ施術に応じた一般的なリスクを記載
- **tags**: 施術名・部位・効果・悩みなどSEO観点で有用なキーワード
- **thumbnail**: 通常 `image_1.jpg`（Before/Afterの1枚目）

### Step 6: 本文を執筆する

画像とキャプションの内容を基に、症例記事として執筆する。

**基本構成：**

```markdown
## この症例について
（2〜3文：どんな悩みに対して何をしたか）

## 施術のポイント
（施術の詳細・工夫・効果の解説）

## 施術詳細
| 施術内容 | 詳細 |
|---------|------|
| 施術名 | ○○ |
| 施術部位 | ○○ |
| 経過 | ○○ |

## こんな方に向いています
- 箇条書きで3〜5項目
```

**執筆ルール：**
- 「。」の後には必ず改行する。同じ行に次の文が続くことは絶対にない
- 医療広告ガイドラインを守る：「必ず」「確実に」などの断定表現を使わない
- 効果は「〜が期待できます」「〜に効果的です」など表現する
- キャプションの情報を正確に反映する（施術名・部位は特に正確に）
- **料金情報は記事本文に記載しない**：料金はSupabaseのDBで一元管理（SSOT）。キャプションに料金が記載されていても転記しないこと
- **製品名（アラガン社ジュビダーム等）はキャプションにあればそのまま記載してOK**（料金ではないため）

### Step 7: ファイルを保存する

対象ディレクトリの `article.md` を保存する（既存がある場合は上書き）。

保存後、以下を確認してユーザーに伝える：
- slug（microCMSに登録する際に使用）
- pillar（どのマッピングルールで判定したか）
- thumbnail（microCMSにアップロードすべき画像）
- instagram_urlフィールドが空の場合、元投稿URLがあれば追記を促す

### Step 8: インデックスを更新する

以下の2ファイルに記事情報を追記する：
- `public/case/CASE_INDEX.md` — 該当部位のテーブルに行を追加
- `public/ARTICLE_INDEX.md` — 症例セクションの記事数を更新

### Step 9: ディレクトリをリネームする

記事作成・microCMS投稿完了後、対象ディレクトリを `{番号}_{記事タイトル}` 形式にリネームする。

例：`001` → `001_ヒアルロン酸注入で整える横顔`

タイトルは日本語で、簡潔に（長い場合は核心部分のみ）。
リネーム済みディレクトリ = 「処理済み」の証跡になる。番号のみのディレクトリ = 未処理。

## microCMS投稿コマンド

article.md完成後、以下のコマンドでmicroCMSに投稿する：

```bash
export $(grep -v '^#' .env.local | grep -v '^image' | xargs) && node scripts/microcms-case-post.js "public/case/{ディレクトリ}"
```

投稿成功後、`public/case/urls.txt` の該当行を `# [完了] {url}` に更新する。

## 注意事項

- **pillar判定は必ず施術名→マッピング表で行う**（施術部位や効果で推測しない）
- 施術名がマッピング表にない場合はユーザーに確認する
- 複数施術の組み合わせ（例: 糸リフト＋ヒアルロン酸）の場合、主たる施術のpillarを採用。判断に迷う場合はユーザーに確認
- published_atの取得方法: `published_at.txt` があればその内容を使う。なければinstaloaderのダウンロードファイル名（`YYYY-MM-DD_HH-MM-SS_UTC`形式）の先頭から抽出する。どちらもなければ本日の日付を使用する
- instagram_urlは空のまま保存してOK（後からユーザーが追記）
- 画像に含まれるテキストは正確に反映する
- CASE_INDEX.mdの更新を忘れないこと
- **1件完了するごとに結果を報告してから次へ進む**（タイトル・slug・pillar・microCMS IDを出力）
- **エラー時（DL失敗・画像なし等）は報告してスキップし、止まらずに次のURLへ進む。ただし空ディレクトリを作らないこと**
- **git操作・開発サーバー起動は行わない**（CLAUDE.mdのOperator Rulesに準拠）
